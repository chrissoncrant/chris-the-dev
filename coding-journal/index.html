Next up I went through the section on width. I learned some stuff here.

Specifically I learned about the difference between width: auto (this is the default value) and width: 100%. When the width of an element is set to 100% it means that the element’s content-box will always be 100% the size of its parent. If horizontal margin is added to the element, then there’s a chance the element may break out of its parent because that margin will be added to the width which is already 100%. This is good to know~

I also learned about some keywords to use for width. In addition to fit-content, which I already knew about, I learned about min-content and max-content. These keywords are interesting because they are intrinsic values, which means the width is not based on the parent (extrinsic), but is based on the width of the children.

With min-content this means the element looks for the longest single word and then makes its width be equal to that word. That longest word becomes the width of the element.

With max-content the width of the element is equal to the width of the widest child. If a header is very long it means the width will be as long as that line is without any line-breaks. This can break outside of a parent.

Both of these methods are interesting ways to shrink wrap content, but the most straight-forward one seems to be fit-content (just have to be sure to add the vendor prefix for Firefox!)

Finished Chapter 10: Prototypes

This is the last chapter in this course.

I spent some time learning about prototype chains awhile back. This chapter helped me better understand what these are, especially in the context of the ‘wiring’ mental model.

Prototypes are mostly relevant when reading property values. If a property is being written, this will occur on the object that the property is being written to, not a prototype.

It is possible to write to an object’s prototype. This is called Prototype Pollution and for the most part should be avoided.

Every newly created object has as its prototype a default Object Prototype which is where all those fancy methods and properties live and are accessible from. The prototype of the Object Prototype is just null. This is the end of the line for every prototype chain.

Like properties, prototype chains are unidirectional. They flow down stream in the form of object chains. A prototype cannot access the properties of the object of which it is a prototype.

Learning: Just JavaScript

Today I went through Chapter 8: Properties.

This expands on the variable mental model of wires connecting to values. Essentially properties behave like wires and just like all wires in this mental model they must be connected to a value.

One difference between properties and variables is that variables are code which point/connect to values. Properties don’t start from code, but start from values (object values). Variables are code connecting to values; properties are values connecting to other values.

Another aspect is that properties are not inside objects, but are simply connections that are grouped together via the object value. The object value is a sort of hub for values and the properties are what connect the values to the hub.

The reason this is helpful is because the values of connected to properties are available to other variables and other object values. They don’t enter the object to gain access, but simply connect via their own wires to these values.

How can something be in two boxes at the same time? This is the problem with the box mental model. The wire mental model is more accurate in explaining the behavior of variables and properties.

I also learned about what the ‘this’ keyword does. The course doesn’t go over ‘this’ but through one of the problems I learned more about it.

//The code below would throw an error because the 'next' property is pointing to the 'president' variable's value before it has been declared. 
                
let president = {
    name: "Goofy",
    next: president
}
                
// Without the 'this' keyword this is accomplished like so:
                
let president = {
    name: "Goofy",
}

president.next = president;
                
//The same thing above can be accomplished with less code by using the 'this' keyword:
                
let president = {
    name: "Goofy",
    next: this,
}

When an object gets more complicated ‘this’ becomes even more useful. Encountering this problem helped me understand how ‘this’ behaves and what it replaces. I now have a better grasp on what ‘this’ is abstracting.

I also learned about how the dot operator works in regards to accessing properties.

If the value to the left of the ‘.’ is undefined or null an error is thrown, otherwise it moves forward.

This makes sense because ‘undefined’ and ‘null’ are primitive values and do not posses any properties in and of themselves.

If a property of an object hasn’t been declared, trying to access it will result in undefined. This does not mean that in doing so a property is created and wired to the value ‘undefined’. In other languages trying to access a property that doesn’t exist would result in an error. In the Universe that is JavaScript, this results in the value ‘undefined’. It’s just a law of this Universe.

Continuing with the introductory module. Continued learning about units by reading Josh’s article.

I found this article informative. It revealed a bad habit I have developed of using pixels for font sizes. This is bad for accessibility reasons. Moving forward I will be using rem units for font-sizing.

He made an interesting case for using ems for vertical margins, which prevents excess code writing. For example: if I wanted my margins for all headers no matter what type to be 0.75x their font-size size I would have to calculate that value for every header if I were using rem units. However if I were to use ems for the margins I could just have one rule with a selector for all headers and set the margin (vertical) to 0.75em and this would take care of it that size, then I would have to calculate that rem value.

The article also made a very interesting case for using rem units for media queries. I am still wrapping my head around this. The example he uses to highlight the usefulness of doing this has to do with a page layout that has a vertical navigation on the left hand side. If a user bumped the default browser font size to 32px and pixels were used in the media query, then the navigation would end up taking up a larger portion of the available space, squeezing the main content.

By using rems for the media query, when the user bumps up the font size, then the media query breakpoints would also change. In this case what would occur is the page would be following the mobile specifications and the navigation would become collapsible. This is pretty neat!

The article was very thorough and offered some interesting workflows and I applied some of the ideas to this website and am looking forward to making these ideas a part of my workflow.

With that said it was also emphasized that there are no hard rules and that the more projects we build the more our intuition will build in regards to determining what kinds of units to use.

He left me with a good mental model for deciding in the form of a simple, yet powerful question: Do I want this to scale up when a user changes a browser’s default font size?

Chapter 6: Meeting Objects and Functions

Objects and functions are both values. These are values that can be summoned (similar to primitive values), but in order for an object value and function value to be summoned it first must be created.

That’s right! Objects and functions are values that we get to create! Joy!

There are many types of objects. Dates, arrays, RegExps, Number, Math, etc.

Objects can be created, but not destroyed. JavaScript has a garbage collection, but when this occurs cannot be guaranteed.

Functions are objects, but with special properties. Functions possess the capabilities of objects, but not necessarily vice versa.

When a variable is wired up to a function value, that function is created at that point in time.

If another variable is created an wired to a function value that looks the same, it does not wire up to the same function value; a new function value is created. To wire up a variable to the same function value, then the variable must point to the function value already wired up to the other variable. (Ex: let oldVariable = function(){}; let newVariable = oldVariable;)

In order for the code within the function body to be run, it must be called using ‘().’ When a function is called it is a call expression. The call expression is answered by JavaScript by running the code, resulting in whatever value is returned.

Writing it out:


let plant = function () {
    return {};
};

let tree = plant();

let flower = plant();

The first line created a variable called ‘plant’ and wires this up to a newly created function value.

The second line creates a new variable called ‘tree’ and wires it up to another newly created object.

That process looks like this:

The plant variable expression results in a function value. This function value enters into the call expression, which results in an object literal expression, which results in a newly created object.

The third line creates a new variable called ‘flower’ and runs the same process above and is wired up to another new object.

After all this code runs there are 3 newly created values! A function value and two distinct object values.

Chapter 7: Equality of Values

There are three kinds of equality: Strict Equality (a === b), Loose Equality (a == b) and Same Value Equality (Object.is(a, b)).

I have not encountered Same Value Equality yet, but this chapter started with it.

Due to the mental models of values that has been set up throughout the course so far, Same Value Equality is easy to grasp. Where I once would have been confused (objects), I am no longer confused due to what I learned in Chapter 5.

Strict Equality almost behaves the same, except for two special cases.

Both cases deal with primitive special numerical cases.

The first one has to do with NaN. With Strict Equality NaN === NaN is always false, whereas Object.is(NaN, NaN) is true.

The second case has to do with -0 and 0. We know that -0 and 0 are individual and unique values and Same Value Equality shows this is the case:

Object.is(-0, 0) will result in false, whereas -0 === 0 will always result in true.

Out of these two special cases the one that will most likely be encountered is the case dealing with NaN. In order to check that a value is NaN we can do one of three things:

Object.is(someVariable, NaN)

Number.isNaN(someVariable)

someVariable !== someVariable

Out in the world, Strict Equality is what I will be using fo the most part, but it is super helpful to be aware fo those two special cases, especially the first.

As for Loose Equality, I got one word: YUCK.

I will not be using it if I can avoid it.

Here is a useful resource which shows all the strange results of Loose Equality: JS Comparison Table

This course doesn’t cover it so much and essentially warns to use with caution.

But again, I am simply going to avoid the use altogether.

One potential useful use case is: variableName == null

This is like this: someVariable === null || someVariable === undefined

But seriously? Is that really that much more code to write? I say no.

Chapter 3: Values and Variables

A number of mental models were established within this chapter.

These mental models build on the mental model of values and are related to variables.

These mental models are:

  • Primitive values are immutable. Object and functions are mutable.
  • Variables are wires; they are associations between the variable name and the value.
  • Variables are not values, but wires connecting to values.
  • Variables are expressions.
  • Variables cannot point to other variables, only to values.
  • Variables are never ‘passed’ into functions; the value the variable points to is what is passed.

Chapter 4: Studying from the Inside

This mental model helps to isolate the focus.

Any programming involves so many layers of abstraction and it can get easy to get lost in these layers. This mental model holds us within the appropriate layer.

We don’t care about how the computer stores the memory and how all that craziness works. It’s just a distraction. What we care about are the qualities, characteristics and governing laws of JavaScript. That’s what we study.

The foundation is: Values are not code. Most values are immutable. Variables are pointers to values, are connected to values through ‘wires.’ Expressions are questions that JavaScript can answer. These answers are in the form of variables.

Chapter 5: The Primitive Values

I am appreciating the order this chapter is introducing these: in order of how many values there are of each primitive data type.

Undefined has one and only one flavor of its value and that is undefined.

Null is similar. Null does have the typeof quirk, but other than that the difference is more human related than machine related. The machine treats them the same. In regards to human it all has to do with intention. Undefined represents unintentionally missing values, whereas null represents intentionally missing values.

Boolean values have two flavors: true and false. There are not multiple trues, nor are there multiple false. However each flavor can have infinite numbers of wires connecting them to different variables.

Numbers. These have so many flavors. However the amount is not limited. Numbers also behave in quirky ways due to their floating-point nature. Their precision of reduces the further we get from 0. There are also special numbers: NaN, -Infinity, and Infinity, which exist to account for special circumstances.

BigInts allow are arbitrary-precision integers. Not fully sure what that means, but it expands JavaScript’s numerical capabilities. These can be created by simply adding an ’n’ to the end of any integer. These are important when it comes to certain areas of finance.

Strings have many flavors as well. Every character and pattern of characters represents a unique flavor of string. Based on our Inside JavaScript mental model, all strings already exist, we do not create them, but essentially summon them. Also strings have special properties that seem similar to those of arrays (objects), but these properties don’t behave exactly the same because strings are primitive and immutable.

Symbols. These are not touched upon very much because they involve learning more about objects and so for now it is enough to know that they are similar to keys and they allows us to store/hide information inside an object and set what parts of the code can access that information.

Thoughts and Review: Atomic Design

Reviewing what I’ve learned so far in the Atomic Design course.

Atoms are the simplest pieces. These are buttons, specific kinds of text (headers, descriptions, excerpts, paragraphs), specific kinds of images (profile pics, hero images, logos)

Molecules are combinations of atoms to form larger functional pieces such as a search input with its search button.

Organisms bring together molecules to form even more complex functional patterns consisting of combinations of atoms, molecules and even other organisms. Example would be a header which contains a navigation organism, a logo atom and a search molecule.

Templates are how these things are brought together within a page. Templates are generic pages with generic content. These begin to give us an idea of how all the organisms come together to form an experience.

Pages bring in the specific content to give a finished look. This is also the stage where we can really begin to test out variations in content, using test cases to wrinkle out any issues.

How this is actually implemented within a design system involves file/directory structure, naming conventions for the files, templating languages, such as Mustache, testing data (can be in the form of JSON), the variations of data (default/filler, best case data, variation scenarios).

This also involves documenting, annotating and providing component lineages. Documenting is about describing what each component, usually in the form of an organism or a molecule, is for, how it is to be used, and other details that can be helpful for designers and developers. Documentation also involves providing a means to view the code related to the component in all the necessary layers (template code, rendered html, css) and to see the finished result.

Annotating is useful and is essentially a way to put the components within the context of an actual page and giving the capability to click on a button to reveal all the documentation related to the component itself.

Documenting the component lineage is a part of documentation as well. This provides information on what atoms, molecules, organisms make up the component, as well as what organisms contain the component. It goes both downward and upward. This is useful for testing, troubleshooting and adjusting.

Thoughts

Spent some time writing some more about the relationships of content, design and structure. I am finding this to be extremely helpful and grounding. All the learning that is to come will be rooted in better, more fertile soil due to having a better understanding of the context web design and development occurs within.

I am coming to learn that I do have certain opinions. For instance, content itself needs to be the boss. Mostly. There is a lot of interplay between content, design, and structure, and yet there is also a main current that flows mostly in a certain direction.

Design and structure should, for the most part be following the lead of content. Editing of content should be occurring outside of what colors and type it will be using and where and how it will be displayed.

With that said, I know there are some books that had to be edited because the book itself needed to be shorter. In these cases structure is an editing factor on the content. But perhaps the structural needs only highlighted missed opportunities for creating more impactful content, forcing hard questions to be asked and darlings to be killed. In this sense structure didn’t necessarily do the editing, but rather created the impetus for improved editing.

In this manner there is always bi-directional flow occurring, but for the most part content is king (or queen) and the flow is from content to design and structure.

Content has a purpose and that purpose informs the style of voice that conveys the words. The purpose and voice are crucial factors when it comes to making design decisions. The color, textures, and typography need to support the purpose and voice of the content. There is simply no reason for this to have any back flow.

Artistically, perhaps it could be fun to start with the limitation of using the color blue and letting that color’s psychological implications generate content. But for the most part this is not the case.

When it comes to structure it gets interesting. There is an obvious connection between content and HTML semantic structure. The structure of the HTML will follow the structure of the content.

But there are other structural aspects that aren’t necessarily dependent on the content itself, but rather are there to make sure that the content is flexibly visible and accessible. This involves external and internal layers.

Externally, this means that the user is able to come into contact with the content. This contact must be able to occur within the majority of circumstances a user may find themselves. These circumstances are inherent to the user (blindness, deafness, motor impairment, etc.) and inherent to the device (screen sizes, interface differences, etc).

Internally, this means that designers, developers and project managers are able to communicate with one another using a shared language that creates internal cohesion. This in turn would greatly benefit the stakeholders for what should be obvious reasons.

The flow between these internal and external structural layers is very much a bi-directional flow. Externally, users encountering the content will have positive and negative experiences. Negative experiences need to be eliminated and positive experiences need to be supported and enhanced.

There is a lot of testing in this and when the factors involved in decreasing negative experiences and increasing positive experiences are discovered, there needs to be efficient ways to implement those discoveries. This is where internal structure in the form of design systems become crucial.
