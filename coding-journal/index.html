Just had a discovery about how align-contents works. This has been a bit vague to me, mainly due to it not being relevant. However today I discovered one area where it comes into play.

Usually a flex container’s height will be set based on its content. However if a flex container has a specific height set and the flex-direction is row with wrapping enabled, when the wrap becomes active, each flex child will fill out the space specified (we’re assuming no overflow happening).

This specified space is the flex-child containment area, however, the actual element living within this containment, doesn’t necessarily have to fill this space. By default align-items is set to stretch, and so in the cross-axis the element is set to stretch the cross space of the containment.

Changing align-items to say flex-start will make the element within the containment its natural size in the cross direction, and then move the element to the start of the containment cell itself.

The key here is the containment cell in the cross direction is always taking up the full allotted space, which is determined by factors such as the height of the flex container and the content of the flex-children. It is the area within which the element can be placed.

Now to change how the containment itself is laid out, we use align-content. Its default value is stretch. If we change this to flex-start, what this does is shrink down the containment to the size of the element itself, and moves them to the cross-axis start of the flex container. In a way, what this does is alters the flex-child containment size.

When the flex-child containment size is no longer stretching, now align-items isn’t effective.

Align-content shrinks the flex-child containments, which means that these containments can be moved within the flex-parent’s containment area.

Align-content isn’t very useful if height is not specified on the flex-parent (assuming flex-direction being row).

————————————

Another thing I learned: there is no justify-content: stretch. In order to have a flex-child stretch to fill in a horizontal direction is to switch the flex-direction altogether. Stretching only occur in the cross-axis direction, not the main axis. Good to know!

Building: Space Tourism Site

Phase two of yesterday’s goals commences today: The link and focus styling analysis.

What I learned:

  • :focus styling a link requires it being in the right order of the other link pseudo-classes. LVFHA. A nice mnemonic for this: LoVe Fears HAte (it’s so precious)
  • If :focus is styled, but no outline declaration is specified within the rule, then the default styling supplied by the browser will be applied in addition to the styles declarations within the rule.
  • If the :focus pseudo class is used, then it will behave a bit differently than the default browser focus styling for links and buttons. The default styling doesn’t style buttons and links that are clicked on with the mouse. The styling only applies to items focused on via the keyboard. With :focus, this changes. The :focus styling will apply to clicked buttons and links and will remain that way until the page is clicked elsewhere. This may not be desirable behavior.
  • Styling using :focus-visible seems to behave like the default styling in that the styling will only be applied when the item is focused on via keyboard commands. The styling remains even when the focused on item is click or entered.
  • The ordering of :focus and :focus-visible matters. It works with the cascade. Later styles will overwrite earlier ones.
  • The :active link pseudo class is a momentary styling and is not useful for indicating a currently selected tab.
  • The :target class is pretty neat in that it is a way to style popups and tooltips.

With all this mind it is time to apply these learnings to the Space Tourism site.

A sloppy summary of this Burger Menu:

We have a button. The good ol’ burger button that we are all familiar with. When the screen is at a certain size it appears and the navigation goes away, and vice versa.

When the button is present, it has a basic functionality of revealing and concealing the navigation.

These are the two visibility concerns.

Then there is the case of focusability. When the button is not showing, we want links to be focusable as they naturally would be. When the button is showing and the menu is closed, we don’t want the links to be focusable. When it is open we do.

These are the conditions that the Burger Menu JS works with. Another word for these conditions is “state”.

Between the user interaction and the resulting changes in the HTML is a state-controlling object. There are two states that need to be controlled: “status” and “enabled”.

The “status” state relates to whether or not the menu is open or closed. This state is only relevant when the button itself is visible.

The “enabled” state relates to whether the button itself is visible. If it is visible, then the “status” state becomes relevant. If it is not visible, then the button is unclickable and so the “status” state is irrelevant.

The initial loading of the page triggers the script to run and set the initial states. The window width will determine the visibility of the button. If the button is initially visible, the menu will be closed (“status” = ‘closed’).

At this point the user can resize the window or click the button and it is these two actions that cause changes to the state object. The state object itself has a handler that will trigger changes in the HTML (attribute values and focusability).

User interactions change the state object, which triggers changes in the HTML. The state object is like a controlling buffer.

The state object is necessary for memory. In order for a thing to open, it must somehow know that it is closed. In order for a thing to be set to be focusable or not, it must know what its surroundings are like. The state object is what conveys this information.

It’s extremely nifty.

Continuing with he Hamburger Menu javascript function, I am led to Proxy objects. I had no idea what these were or how they worked. I now know much more, not only about these special objects, but also about what getters and setters are in general.

Proxy objects are, to me, like secretary objects. They are a layer between the object and the outside world. Almost like how an API endpoint is between the data and the request being made. Proxy objects wrap the “target” object and any calls made to the proxy will go through the specified handler object.

There are many methods that can be specified on the handler object. The most used are probably the getters and setters. Whenever a read request is made to the proxy object, they will be “intercepted” within the get “trap”. Whenever a write request is made on the proxy object these will be intercepted by the set trap.

The calc() function is interesting. It was confusing to me at first how to apply units. For example if I wanted to convert 3px to rems I know I would divide 3 by 16, which give 0.1875 and I can apply ‘rems’ unit to that. But what if I wanted to use the calc function? How would I apply the rem units?

Well, it would look like this: font-size: calc(3rem / 16);

This would essentially give me 0.1875rems.

But at a glance it can be confusing. What this calc function is not doing is first calculating what 3rems is and then dividing by 16, which would give 3rems as the result. What it is doing is calculating what 3 / 16 is, then applying the rem unit to the answer.

This is a syntactical behavior of the calc function.

Learning: CSS for JavaScript Developers

Almost finished module 2. Went over how to hide things via display: none, visibility: hidden, and the opacity property.

This section discussed the differences between these. The display: none declaration removes an element completely from the DOM and it becomes truly inaccessible.

The visibility: hidden declaration makes the element invisible, but does not remove it from flow. Its box with all its box properties are still there. This can be useful for selectively revealing items, such as answers to questions.

Interestingly, with this property one can set a parent element to be visibly hidden, and then selectively set children to be visible.

This property also removes items from being clickable, selectable, and focusable.

The last property, opacity, works similarly to visibility: hidden in that the element remains in flow. However, a key difference is that the element is still clickable, selectable and focusable which can create weird user experiences for people using screen readers.

The most important section of this section, I think, is how to visually hide things for screen users, while keeping them “visible” for screen-reader users. This is important for items like buttons that only contain icons.

There are many ways to do this. One method is to set the aria-label: “Button Label Here” attribute, but this sometimes works and sometimes doesn’t depending on context.

The method described by Josh involves a CSS utility class called ‘screen-reader-only’ or ‘sr-only’. What this does is it takes the element and moves it out of the way through clever declarations so it is not visible to screen users, yet will be ‘visible’ for screen reader users.

Learning: CSS for JavaScript Developers

Continued with Module 2. Finished the sticky positioning section.

Sticky positioning is definitely an interesting declaration. It is tied with scrolling. Without scrolling sticky will do nothing.

It has 2 requirements: position: sticky and at least one top/left/bottom/right value declared, which assigns where the element will stick.

If we set the top value to 0, then what this means is that the element will stick to directly onto the parent’s top edge. If the page scrolls down, then the stickiness occurs at the viewport’s edge until the bottom edge of the parent container is reached. The sticky element cannot leave the boundary of its parent, except with negative top/left/bottom/right values.

Because sticky positioning is tied to scrolling, this means that setting an overflow property on any of the sticky elements ancestors will impact how the element sticks. If the parent has overflow auto, hidden, or scroll set, but doesn’t have a fixed height set along with overflowing content, then it will appear as if sticky positioning is “not working”. It’s a similar situation that occurs if the parent doesn’t have overflow set, but also isn’t tall (or wide) enough.

Chapter 6: Meeting Objects and Functions

Objects and functions are both values. These are values that can be summoned (similar to primitive values), but in order for an object value and function value to be summoned it first must be created.

That’s right! Objects and functions are values that we get to create! Joy!

There are many types of objects. Dates, arrays, RegExps, Number, Math, etc.

Objects can be created, but not destroyed. JavaScript has a garbage collection, but when this occurs cannot be guaranteed.

Functions are objects, but with special properties. Functions possess the capabilities of objects, but not necessarily vice versa.

When a variable is wired up to a function value, that function is created at that point in time.

If another variable is created an wired to a function value that looks the same, it does not wire up to the same function value; a new function value is created. To wire up a variable to the same function value, then the variable must point to the function value already wired up to the other variable. (Ex: let oldVariable = function(){}; let newVariable = oldVariable;)

In order for the code within the function body to be run, it must be called using ‘().’ When a function is called it is a call expression. The call expression is answered by JavaScript by running the code, resulting in whatever value is returned.

Writing it out:


let plant = function () {
    return {};
};

let tree = plant();

let flower = plant();

The first line created a variable called ‘plant’ and wires this up to a newly created function value.

The second line creates a new variable called ‘tree’ and wires it up to another newly created object.

That process looks like this:

The plant variable expression results in a function value. This function value enters into the call expression, which results in an object literal expression, which results in a newly created object.

The third line creates a new variable called ‘flower’ and runs the same process above and is wired up to another new object.

After all this code runs there are 3 newly created values! A function value and two distinct object values.

Chapter One

This chapter is all about revealing the power of mental models. They exist whether we are aware of them or not. Becoming aware of them is important and foundational in being the best coder we can be.

The chapter wraps mental models into the reality of the two speeds at which we think: fast and slow.

Fast thinking involves patterned habitual, thinking and requires little energy.

Slow thinking involves the ability to plan out, re- and de-structure, and take apart a process piece by piece to understand what is happening in order to make what we want to happen happen.

Having accurate mental models is important for both fast thinking and slow thinking.

Accurate mental models lead to less bugs and more efficient debugging.

It is often the case that bugs arise when our fast thinking is based on inaccurate mental models. These bugs require us to enter into slow thinking mode.

When we enter the more laborious slow thinking mode we can, if we are attentive, expose the flaws in our mental models as our mental model is what we are using to step through the process piece by piece.

With the right mentality, bugs become opportunities for growth.

Chapter Two

This chapter was about values.

The mental model of this chapter is: Values are not code.

The concept that values are separate from code was one I haven’t encountered, but I think it is an important one. Our code is what moves values around and utilizes values.

Values however are not dependent upon code. They exist whether there is code or not. They are the raw material waiting to be used. Like granite. It is there whether or not it is “programmed” into being a counter top.

The elucidation of expressions was helpful as well. Expressions are questions that JavaScript can answer and the answer will always be in the form of a value.

Values themselves come in 9 and only 9 types. There are two broad categories of values: primitive values and then objects and functions. It was made clear that functions are essentially types of objects, but they are special enough to warrant a pseudo-separation from objects.

Primitive values are the most raw material and consist of numbers, strings, booleans, null, undefined, symbols, and bigints.

Then there are objects and functions which are values, but can be interacted with uniquely using code.

Thoughts

Spent some time writing some more about the relationships of content, design and structure. I am finding this to be extremely helpful and grounding. All the learning that is to come will be rooted in better, more fertile soil due to having a better understanding of the context web design and development occurs within.

I am coming to learn that I do have certain opinions. For instance, content itself needs to be the boss. Mostly. There is a lot of interplay between content, design, and structure, and yet there is also a main current that flows mostly in a certain direction.

Design and structure should, for the most part be following the lead of content. Editing of content should be occurring outside of what colors and type it will be using and where and how it will be displayed.

With that said, I know there are some books that had to be edited because the book itself needed to be shorter. In these cases structure is an editing factor on the content. But perhaps the structural needs only highlighted missed opportunities for creating more impactful content, forcing hard questions to be asked and darlings to be killed. In this sense structure didn’t necessarily do the editing, but rather created the impetus for improved editing.

In this manner there is always bi-directional flow occurring, but for the most part content is king (or queen) and the flow is from content to design and structure.

Content has a purpose and that purpose informs the style of voice that conveys the words. The purpose and voice are crucial factors when it comes to making design decisions. The color, textures, and typography need to support the purpose and voice of the content. There is simply no reason for this to have any back flow.

Artistically, perhaps it could be fun to start with the limitation of using the color blue and letting that color’s psychological implications generate content. But for the most part this is not the case.

When it comes to structure it gets interesting. There is an obvious connection between content and HTML semantic structure. The structure of the HTML will follow the structure of the content.

But there are other structural aspects that aren’t necessarily dependent on the content itself, but rather are there to make sure that the content is flexibly visible and accessible. This involves external and internal layers.

Externally, this means that the user is able to come into contact with the content. This contact must be able to occur within the majority of circumstances a user may find themselves. These circumstances are inherent to the user (blindness, deafness, motor impairment, etc.) and inherent to the device (screen sizes, interface differences, etc).

Internally, this means that designers, developers and project managers are able to communicate with one another using a shared language that creates internal cohesion. This in turn would greatly benefit the stakeholders for what should be obvious reasons.

The flow between these internal and external structural layers is very much a bi-directional flow. Externally, users encountering the content will have positive and negative experiences. Negative experiences need to be eliminated and positive experiences need to be supported and enhanced.

There is a lot of testing in this and when the factors involved in decreasing negative experiences and increasing positive experiences are discovered, there needs to be efficient ways to implement those discoveries. This is where internal structure in the form of design systems become crucial.
