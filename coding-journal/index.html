Learning: Frontend Masters/The New Hard Parts

More thinking on async JS.

I have some code that needs to run when something happens. I don’t know when that something will happen, but when it does happen I need some other things to happen. But only when the first something happens.

What the Fetch API does is sets up a listener and assigns functions to run in a specific order.

The things that need to happen occur within the browser itself. The callstack() is free, not blocked by this process. The process finishes, that has a result, this result triggers the other functions to run.

It’s that whole setting up a listener thing. It’s that whole moving long processes outside of JavaScript as much as possible and tying JavaScript code to run. Asynchronous happenings within a synchronous environment.

————————————

I am seeing now that for the most part, we don’t want any tasks that take a long time to occur within JavaScript. We want to keep that callstack() open and responsive to do what it needs to do.

The most common form of long-running processes that we want to keep out of JS land is the HTTP request. Depending on the request and distance to server, server traffic handling capacity (bandwidth?), the time it takes is highl variable. HTTP requests are always open ended where time is concerned.

This process occurs outside JavaScript, within the runtime environment. The runtime (browser, nodejs, etc) is packaging, parsing, serializing/deserializing, making the network connect, sending/receiving the request/response. This task takes a certain amount of time. While its doing its thing, we don’t want JavaScript to be hung up waiting for the result.

Fetch is an API that allows us to make HTTP requests (utilize the runtime’s network feature), attach functions and error handling to the response, and when the response is completed, auto-run those functions based on the response. And ultimately it allows us to keep the callstack() free and open while this happens.

Learning: Frontend Masters/The New Hard Parts

Going over async JS again today.

Async JS is a capability provided not by JS itself, but the runtime environment.

The common runtime environment is the Browser. Browsers run on computers. Browsers can interact with computers in ways JS cannot. Computers are not single-threaded. JS is single-threaded.

Through ‘facade’ functions, which are labels in JS, we can interact with Web APIs, which give us capabilities beyond JS.

The two biggest features that Browsers and the other runtime, Node, give us are Timers and Network features. Both of these allow us to essentially bump long-running, blocking code in the background and give us the opportunity to invite code back into the Thread of Execution at a later time, assigning tasks/callbacks to run upon completion of the blocking code.

In order for this to work, there are two other runtime features: callback queue and event loop.

The callback queue is where callbacks related to Timer and Network features are added. These callbacks get queued and are added to the callstack (callstack is JS) via the event loop. In a way the event loop is what connects the non-JS task results back into the JS world.

The event loop monitors that callstack and once the callstack is empty, the first callback in the callback queue is added to the callstack.

————————————

Breaking down what setTimeout does a bit further. When setTimeout is encountered by JS, the Timer feature of the runtime is invoked, a callback and an optional duration are passed as arguments. After the TImer is invoked, JS moves onto the next line.

If there is a duration for the Timer, that duration sets the time at which the callback is added to the queue.

In a sense, setTimeout is saying “run this specific block of code, but do so at a later time; don’t block the rest of the code in the current Thread of Execution from running.” In another sense, it bumps the code to the end of the line. This allows other important code to run to its conclusion before the blocking code is executed.

Reviewing more conditionals as well as logical and ternary operators. I thought this would be a good time for me to review the short-circuiting effects of logical operators.

When using an ‘and’ (&&) operator, if the first evaluated value is truthy, then the second value is returned. If the first value is falsy, then that falsy value is returned.

With the ‘or’ operator, it is the opposite. Exactly. If the first evaluated value is truthy, that value is returned. If the first value is falsy, then the second value is returned.

This is generally something that is still a bit fuzzy in my head to keep track of what exactly will happen. It is, however, easy to check and verify.

As a whole, the ‘or’ operator evaluates to truthy if at least one of the values is true. The ‘and’ operator evaluates to truthy if both the values are truthy.

It seems to me that these are two different things. Short-circuiting is used for assigning variables, whereas the evaluation of the whole logical operator expression is used within the parentheses of conditionals.

On closer inspection, these aren’t separate at all. It’s just a matter of recognizing that the logical operators are a sort of two part expression.

Within the parens of the conditional, if we are using the ‘and’ operator, if the first value is falsy, it makes sense to return that value, then that first value is evaluated and, on its own, will return falsy. With the ‘or’ operator, if the first value is falsy, then it makes sense to return the second value and have that value be what decides the truthiness/falsiness of the overall expression.

There are 5 things that were mentioned in making an accessible modal, which is not a straight-forward thing.

The last thing is to make the modal friendly to screen readers, it needs to be annotated that a user has been moved into a modal. This is accomplished by adding HTML attributes. These attributes are:

  • aria-modal=“true/false”
  • role=“dialog”
  • tabindex=“-1” (when aria-modal=“true”)
  • aria-label=“title-of-modal”

I ended today’s adventure in Josh’s course by going over breakpoints. I can dig Josh’s philosophy of setting breakpoints within dead zones between device sizes (phones vs. tablets vs. laptops vs. desktops) versus setting them at device specific sizes (iPhone 12, etc).

Generally different devices come in clusters of sizes. Instead of setting a breakpoint at a device specific size of say 375px, we would add in some wiggle room and make it 550px. This wiggle room would account for similar devices (bigger and smaller). Basically, similar devices should be falling within the middle of the breakpoint, rather than setting the edges of the breakpoints.

The breakpoints Josh uses are:

  • 0-550px — Mobile
  • 550-1100px — Tablet
  • 1100-1500px — Laptop
  • 1500+px — Desktop

Learning: CSS for JavaScript Developers

Spending some more time becoming familiar with these hidden concepts that make up the functionality of the beloved flexbox.

First there’s the axes: main and cross. The main axis is the axis where stuff gets applied. By ‘stuff’ I mean flex-grow, flex-shrink, flex-basis… all these work in the main axis direction.

I also found it interesting that there is no justify-content: stretch. Align-items property has the default value of ‘stretch’. Align-items always deals with the cross axis, never the main axis, which is justify-content’s axis. It makes sense that there is no ‘stretch’ value for the main axis because it is along the main axis that we generally want to have things set next to each other or stack, depending on the flex-direction. Just thought that was interesting.

The sizes: there’s the hypothetical size, which is the natural size the element would be and then there’s the minimum-content size. The landscape that exists between the hypothetical size and the minimum-content size is the realm of flex-shrink.

The landscape between the available unused space and the hypothetical space is the realm of flex-grow.

Without specifying widths (or heights depending on the flex-direction; we will assume the default row direction for now), the minimum width any flex-child can be is the minimum-content width, which is the width of the padding plus the width of the longest word.

Setting a width comes in 3 flavors: minimum, static, maximum. Setting a minimum-width will overwrite flex’s minimum-content limitation to be that value. Essentially any minimum-width value set will become the minimum-content width in terms of how flex-shrink works.

Setting a max-width is capping the width that the flex-child can grow. It will grow up to that size. That is straight forward.

Setting a static width sets the hypothetical size. Statically sized elements can still grow if given the value, and they can also shrink.

Flex-basis will always win over width/height. No matter where the width declarations are made in the cascade. Flex-basis sets the hypothetical size.

When flex-grow is applied evenly, the distribution of space for the elements will depend on the hypothetical space. If element A is 100px wider than element B, when both elements grow to fill the space, the difference in width between the two will still be 100px. They will both be wider, but the difference in the hypothetical width will be retained.

Flex-wrap comes into play as soon as the hypothetical size limit is reached.

This morning I have been learning more about the subtleties of how flexbox works. I feel I have learned quite a lot so far.

What I have learned is first that there are two sizes to know about. There’s the minimum-content size and there’s the hypothetical size.

The minimum-content size is essentially the minimum size a flex-child will shrink to no matter what. This is the width of the longest word, plus whatever inline padding is applied.

The hypothetical size is the size the element would naturally be. The text is not wrapped and the padding is applied.

Flex-shrink and flex-grow are properties that deal with either side of the hypothetical size of the element.

The easiest to understand for me is flex-grow. We have a flex-child that naturally takes up so much space, and in the main axis direction there is so much space left over. Flex-grow deals with this leftover space and tells the element how much of that leftover space it can take up. The value of the property determines what proportion of that available space is taken up.

Flex-shrink is different. First, if a flex-child only has one word, then flex-shrink will do nothing because technically that element is already at its minimum-content size. Or, to put it another way, its minimum-content size and its hypothetical size are equal.

If a flex-child has two words, then the flex-shrink distance comes into play the moment that second word wraps to the next line. Now there’s a difference between the hypothetical size and the minimum-content size. The distance that flex-shrink applies to equates to the difference between the current size of the flex-child element and the minimum-content size.

The flex-shrink value and shrink rate has nothing to do with the rate at which an element goes from its flex-grown size to its hypothetical size. It only applies to the difference between hypothetical size and the minimum-content size. As soon as the hypothetical size is breached, then flex-shrink comes into play.

Just had a discovery about how align-contents works. This has been a bit vague to me, mainly due to it not being relevant. However today I discovered one area where it comes into play.

Usually a flex container’s height will be set based on its content. However if a flex container has a specific height set and the flex-direction is row with wrapping enabled, when the wrap becomes active, each flex child will fill out the space specified (we’re assuming no overflow happening).

This specified space is the flex-child containment area, however, the actual element living within this containment, doesn’t necessarily have to fill this space. By default align-items is set to stretch, and so in the cross-axis the element is set to stretch the cross space of the containment.

Changing align-items to say flex-start will make the element within the containment its natural size in the cross direction, and then move the element to the start of the containment cell itself.

The key here is the containment cell in the cross direction is always taking up the full allotted space, which is determined by factors such as the height of the flex container and the content of the flex-children. It is the area within which the element can be placed.

Now to change how the containment itself is laid out, we use align-content. Its default value is stretch. If we change this to flex-start, what this does is shrink down the containment to the size of the element itself, and moves them to the cross-axis start of the flex container. In a way, what this does is alters the flex-child containment size.

When the flex-child containment size is no longer stretching, now align-items isn’t effective.

Align-content shrinks the flex-child containments, which means that these containments can be moved within the flex-parent’s containment area.

Align-content isn’t very useful if height is not specified on the flex-parent (assuming flex-direction being row).

Building: Space Tourism Site

Phase two of yesterday’s goals commences today: The link and focus styling analysis.

What I learned:

  • :focus styling a link requires it being in the right order of the other link pseudo-classes. LVFHA. A nice mnemonic for this: LoVe Fears HAte (it’s so precious)
  • If :focus is styled, but no outline declaration is specified within the rule, then the default styling supplied by the browser will be applied in addition to the styles declarations within the rule.
  • If the :focus pseudo class is used, then it will behave a bit differently than the default browser focus styling for links and buttons. The default styling doesn’t style buttons and links that are clicked on with the mouse. The styling only applies to items focused on via the keyboard. With :focus, this changes. The :focus styling will apply to clicked buttons and links and will remain that way until the page is clicked elsewhere. This may not be desirable behavior.
  • Styling using :focus-visible seems to behave like the default styling in that the styling will only be applied when the item is focused on via keyboard commands. The styling remains even when the focused on item is click or entered.
  • The ordering of :focus and :focus-visible matters. It works with the cascade. Later styles will overwrite earlier ones.
  • The :active link pseudo class is a momentary styling and is not useful for indicating a currently selected tab.
  • The :target class is pretty neat in that it is a way to style popups and tooltips.

With all this mind it is time to apply these learnings to the Space Tourism site.

A sloppy summary of this Burger Menu:

We have a button. The good ol’ burger button that we are all familiar with. When the screen is at a certain size it appears and the navigation goes away, and vice versa.

When the button is present, it has a basic functionality of revealing and concealing the navigation.

These are the two visibility concerns.

Then there is the case of focusability. When the button is not showing, we want links to be focusable as they naturally would be. When the button is showing and the menu is closed, we don’t want the links to be focusable. When it is open we do.

These are the conditions that the Burger Menu JS works with. Another word for these conditions is “state”.

Between the user interaction and the resulting changes in the HTML is a state-controlling object. There are two states that need to be controlled: “status” and “enabled”.

The “status” state relates to whether or not the menu is open or closed. This state is only relevant when the button itself is visible.

The “enabled” state relates to whether the button itself is visible. If it is visible, then the “status” state becomes relevant. If it is not visible, then the button is unclickable and so the “status” state is irrelevant.

The initial loading of the page triggers the script to run and set the initial states. The window width will determine the visibility of the button. If the button is initially visible, the menu will be closed (“status” = ‘closed’).

At this point the user can resize the window or click the button and it is these two actions that cause changes to the state object. The state object itself has a handler that will trigger changes in the HTML (attribute values and focusability).

User interactions change the state object, which triggers changes in the HTML. The state object is like a controlling buffer.

The state object is necessary for memory. In order for a thing to open, it must somehow know that it is closed. In order for a thing to be set to be focusable or not, it must know what its surroundings are like. The state object is what conveys this information.

It’s extremely nifty.

Learning: CSS for JavaScript Developers

Almost finished module 2. Went over how to hide things via display: none, visibility: hidden, and the opacity property.

This section discussed the differences between these. The display: none declaration removes an element completely from the DOM and it becomes truly inaccessible.

The visibility: hidden declaration makes the element invisible, but does not remove it from flow. Its box with all its box properties are still there. This can be useful for selectively revealing items, such as answers to questions.

Interestingly, with this property one can set a parent element to be visibly hidden, and then selectively set children to be visible.

This property also removes items from being clickable, selectable, and focusable.

Sticky positioning is definitely an interesting declaration. It is tied with scrolling. Without scrolling sticky will do nothing.

It has 2 requirements: position: sticky and at least one top/left/bottom/right value declared, which assigns where the element will stick.

If we set the top value to 0, then what this means is that the element will stick to directly onto the parent’s top edge. If the page scrolls down, then the stickiness occurs at the viewport’s edge until the bottom edge of the parent container is reached. The sticky element cannot leave the boundary of its parent, except with negative top/left/bottom/right values.

Because sticky positioning is tied to scrolling, this means that setting an overflow property on any of the sticky elements ancestors will impact how the element sticks. If the parent has overflow auto, hidden, or scroll set, but doesn’t have a fixed height set along with overflowing content, then it will appear as if sticky positioning is “not working”. It’s a similar situation that occurs if the parent doesn’t have overflow set, but also isn’t tall (or wide) enough.

If another variable is created an wired to a function value that looks the same, it does not wire up to the same function value; a new function value is created. };

let tree = plant();

let flower = plant();

The second line creates a new variable called ‘tree’ and wires it up to another newly created object.

The plant variable expression results in a function value. This function value enters into the call expression, which results in an object literal expression, which results in a newly created object.

Thoughts and Review: Atomic Design

Reviewing what I’ve learned so far in the Atomic Design course.

Atoms are the simplest pieces. These are buttons, specific kinds of text (headers, descriptions, excerpts, paragraphs), specific kinds of images (profile pics, hero images, logos)

Molecules are combinations of atoms to form larger functional pieces such as a search input with its search button.

Organisms bring together molecules to form even more complex functional patterns consisting of combinations of atoms, molecules and even other organisms. Example would be a header which contains a navigation organism, a logo atom and a search molecule.

Templates are how these things are brought together within a page. Templates are generic pages with generic content. These begin to give us an idea of how all the organisms come together to form an experience.

Pages bring in the specific content to give a finished look. This is also the stage where we can really begin to test out variations in content, using test cases to wrinkle out any issues.

How this is actually implemented within a design system involves file/directory structure, naming conventions for the files, templating languages, such as Mustache, testing data (can be in the form of JSON), the variations of data (default/filler, best case data, variation scenarios).

This also involves documenting, annotating and providing component lineages. Documenting is about describing what each component, usually in the form of an organism or a molecule, is for, how it is to be used, and other details that can be helpful for designers and developers. Documentation also involves providing a means to view the code related to the component in all the necessary layers (template code, rendered html, css) and to see the finished result.

Annotating is useful and is essentially a way to put the components within the context of an actual page and giving the capability to click on a button to reveal all the documentation related to the component itself.

Documenting the component lineage is a part of documentation as well. This provides information on what atoms, molecules, organisms make up the component, as well as what organisms contain the component. It goes both downward and upward. This is useful for testing, troubleshooting and adjusting.

Design and structure should, for the most part be following the lead of content. Editing of content should be occurring outside of what colors and type it will be using and where and how it will be displayed.

With that said, I know there are some books that had to be edited because the book itself needed to be shorter. In these cases structure is an editing factor on the content. But perhaps the structural needs only highlighted missed opportunities for creating more impactful content, forcing hard questions to be asked and darlings to be killed. In this sense structure didn’t necessarily do the editing, but rather created the impetus for improved editing.

In this manner there is always bi-directional flow occurring, but for the most part content is king (or queen) and the flow is from content to design and structure.

Content has a purpose and that purpose informs the style of voice that conveys the words. The purpose and voice are crucial factors when it comes to making design decisions. The color, textures, and typography need to support the purpose and voice of the content. There is simply no reason for this to have any back flow.

Artistically, perhaps it could be fun to start with the limitation of using the color blue and letting that color’s psychological implications generate content. But for the most part this is not the case.

When it comes to structure it gets interesting. There is an obvious connection between content and HTML semantic structure. The structure of the HTML will follow the structure of the content.

But there are other structural aspects that aren’t necessarily dependent on the content itself, but rather are there to make sure that the content is flexibly visible and accessible. This involves external and internal layers.

Externally, this means that the user is able to come into contact with the content. This contact must be able to occur within the majority of circumstances a user may find themselves. These circumstances are inherent to the user (blindness, deafness, motor impairment, etc.) and inherent to the device (screen sizes, interface differences, etc).

Internally, this means that designers, developers and project managers are able to communicate with one another using a shared language that creates internal cohesion. This in turn would greatly benefit the stakeholders for what should be obvious reasons.

The flow between these internal and external structural layers is very much a bi-directional flow. Externally, users encountering the content will have positive and negative experiences. Negative experiences need to be eliminated and positive experiences need to be supported and enhanced.

There is a lot of testing in this and when the factors involved in decreasing negative experiences and increasing positive experiences are discovered, there needs to be efficient ways to implement those discoveries. This is where internal structure in the form of design systems become crucial.
